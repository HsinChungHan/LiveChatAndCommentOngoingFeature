    實作與 TDD 差異

    1.1 Response 格式改進（優於 TDD）

    實作改進： - API 實際返回 直接的 DTO/Array，而非包裝在 bizCode/data 結構中 - 例如： - getBatchCommentInfo: 返回 [CommentMetaInfoDTO]（直接 Array） - getCommentMeta: 返回 CommentMetaDataDTO（直接 DTO） - publishComment: 返回 CommentDTO（直接 DTO） - toggleLike: 返回 null（raw null）

    TDD 原設計： 

    public struct BatchCommentInfoResponseDTO: Decodable, Sendable {
    public let bizCode: Int
    public let innerMsg: String?
    public let message: String?
    public let data: [CommentMetaInfoDTO]
}

    實際實作： 

    // 直接返回 Array，無包裝
public func getBatchCommentInfo(refIdList: [String]) async throws -> [CommentMetaInfoDTO] {
    let endpoint = PrematchCommentEndpoint.getBatchCommentInfo(refIdList: refIdList)
    let response: [CommentMetaInfoDTO] = try await apiClient.request(endpoint)
    return response
}

    評估：✅ 應該更新 TDD - 實作反映了實際 API 行為，比 TDD 更準確。

    1.2 toggleLike 返回值處理

    實作改進： - API 返回 null（raw null），而非包裝的 DTO - 實作使用 NullableResponse 來處理 null 值

    TDD 原設計： 

    public func toggleLike(commentId: Int64) async throws -> CommentDTO?

    實際實作： 

    public func toggleLike(commentId: Int64) async throws {
    let endpoint = PrematchCommentEndpoint.toggleLike(commentId: commentId)
    let _: NullableResponse = try await apiClient.request(endpoint)
}

    評估：✅ 應該更新 TDD - 實作更符合實際 API 行為。

    1.3 publishComment 方法拆分

    實作改進： - 將 publishComment 拆分為兩個方法： - publishComment(refId:, content:) - 發布頂層評論 - publishReply(refId:, content:, parentId:, tagUserId:, sharedBetsMeta:) - 回覆評論

    TDD 原設計： 

    func publishComment(refId: String, content: String, parentId: Int64?, sharedBetsMeta: SharedBetsMetadata?, tagUserId: String?) async throws -> CommentDTO

    實際實作： 

    public func publishComment(refId: String, content: String) async throws -> CommentDTO
public func publishReply(refId: String, content: String, parentId: Int64, tagUserId: String, sharedBetsMeta: SharedBetsMetadata?) async throws -> CommentDTO

    評估：✅ 應該更新 TDD - 拆分後的設計更清晰，區分了頂層評論和回覆的不同業務邏輯。

    public var method: String {
    switch self {
    case .getChatroomInfo, .getHistoricalMessages:
        return FComHTTPMethod.get.rawValue
    case .getBatchCount, .sendMessage, .joinChatroom, .leaveChatroom, .bulkLeaveChatroom:
        return FComHTTPMethod.post.rawValue
    }
}

public var encoding: APIEncoding {
    switch self {
    case .getChatroomInfo, .getHistoricalMessages:
        return .url
    case .getBatchCount, .sendMessage, .joinChatroom, .leaveChatroom, .bulkLeaveChatroom:
        return .json
    }
}

    實作與 TDD 差異

    4.1 檔案結構差異

    TDD 期望： 

    Sources/LiveChat/Services/API/Chat/
  ├── ChatAPI+RepositoryProtocol.swift  # Repository Protocol
  ├── ChatAPI+Repository.swift          # Repository 實作（actor）
  └── ChatAPI+LiveChat.swift            # DTO → Domain Model 轉換

    實際實作： 

    MatchChat/Sources/MatchChat/
  ├── Services/API/Chat/
  │   ├── ChatAPI+RepositoryProtocol.swift  # API Repository Protocol (API 層)
  │   └── ChatAPI+Repository.swift          # API Repository (API 層)
  ├── Services/Repository/LiveChat/
  │   ├── LiveChatRepositoryProtocol.swift  # Domain Repository Protocol
  │   └── LiveChatRepository.swift          # Domain Repository (使用 Client)
  └── Models/LiveChat/
      └── LiveChat+ChatAPI.swift            # DTO → Domain Model 轉換

    評估：⚠️ 架構差異 - TDD 的 ChatAPI+Repository 是指 API 層 Repository，而實作中增加了 Domain 層 Repository。這是更好的分層架構： - API Repository（ChatAPI.ChatRepository）：處理 HTTP 請求，返回 DTOs - Domain Repository（LiveChatRepository）：使用 Client，返回 Domain Models

    4.2 WebSocket 訂閱管理

    實作： 

    public func subscribeWebSocket(chatroomId: String) async throws -> AsyncStream<LiveChat.Message> {
    let webSocketStream = webSocketClient.subscribe(chatroomId: chatroomId)

    return AsyncStream { continuation in
        Task {
            for await webSocketMessage in webSocketStream {
                // 過濾黑名單用戶
                let userId = webSocketMessage.data.jsonBody.postUserId
                if await isBlacklisted(userId: userId) {
                    continue
                }

                let domainMessage = webSocketMessage.data.toDomainMessage()
                continuation.yield(domainMessage)
            }
            continuation.finish()
        }
    }
}

    評估：✅ 實作優於 TDD - 實作包含黑名單過濾邏輯，比 TDD 更完善。

    設計改進記錄

    1. API Response 格式優化

    改進: API 實際返回直接 DTO/Array，而非包裝在 bizCode/data 中

    優點: - 減少不必要的包裝層 - 簡化 Decoding 邏輯 - 更符合實際 API 行為

    影響範圍: - PrematchCommentAPI+Repository.swift - ChatAPI+Repository.swift

    向後兼容性: ✅ 無影響（這是新實作）

    2. publishComment 方法拆分

    改進: 將 publishComment 拆分為 publishComment 和 publishReply

    優點: - 明確區分頂層評論和回覆 - 簡化參數（publishComment 不需要 parentId 等） - 更符合實際業務邏輯

    影響範圍: - PrematchCommentAPI+Repository.swift - PrematchCommentClient.swift - PrematchCommentRepository.swift

    向後兼容性: ✅ 無影響（這是新實作）

    3. 兩層 Repository 架構

    改進: 分為 API Repository 和 Domain Repository

    優點: - 更清晰的職責分離 - API Repository 處理 HTTP 請求，返回 DTOs - Domain Repository 使用 Client，返回 Domain Models - 符合 Clean Architecture 原則

    影響範圍: - ChatAPI+Repository.swift (API 層) - LiveChatRepository.swift (Domain 層) - PrematchCommentRepository.swift (Domain 層)

    向後兼容性: ✅ 無影響（這是新實作）

    4. NullableResponse 處理 null 返回值

    改進: 創建 NullableResponse struct 來處理 API 返回 null 的情況

    優點: - 正確處理 API 返回 null 的情況 - 避免 Decoding 錯誤 - 提供清晰的語意

    影響範圍: - PrematchCommentAPI+Repository.swift (toggleLike) - ChatAPI+Repository.swift (joinChatroom, leaveChatroom, bulkLeaveChatroom)

    向後兼容性: ✅ 無影響（這是新實作）
