    Date: 2025-12-09
    Reason: 實作改進優於原 TDD 設計，更新文檔以反映實際實作

    更新摘要

    本次更新涵蓋了 4 個 TDD tickets，主要是根據實際 API 行為和架構改進來更新文檔：

    Ticket 更新內容 原因
    TDD-010 API Response 格式、方法簽名 實際 API 行為與規劃不同
    TDD-011 HTTP Method、Null Response 處理 實際 API 規格修正
    TDD-031 Repository 架構說明 採用更好的兩層架構
    架構文檔 統一 Package 結構說明 記錄設計決策

    詳細變更內容

    1. TDD-010: PrematchCommentAPI

    變更 1.1: API Response 格式優化

    變更前： 

    public func getBatchCommentInfo(refIdList: [String]) async throws -> [CommentMetaInfoDTO] {
    let endpoint = PrematchCommentEndpoint.getBatchCommentInfo(refIdList: refIdList)
    let response: BatchCommentInfoResponseDTO = try await apiClient.request(endpoint)
    return response.data
}

    變更後： 

    public func getBatchCommentInfo(refIdList: [String]) async throws -> [CommentMetaInfoDTO] {
    let endpoint = PrematchCommentEndpoint.getBatchCommentInfo(refIdList: refIdList)
    // ⚠️ API 直接返回 Array，無包裝層
    return try await apiClient.request(endpoint)
}

    原因： - 實際 API 返回直接的 DTO/Array，而非包裝在 bizCode/data 結構中 - 減少不必要的包裝層，簡化 Decoding 邏輯

    影響範圍： - getBatchCommentInfo - getCommentMeta - getCommentsByPopular - getCommentsByNewest - getReplies - publishComment

    刪除的 DTOs（未使用）： - BatchCommentInfoResponseDTO - CommentMetaResponseDTO - CommentListResponseDTO - CommentResponseDTO - ToggleLikeResponseDTO

    變更 1.2: publishComment 方法拆分

    變更前： 

    func publishComment(
    refId: String, 
    content: String, 
    parentId: Int64?, 
    sharedBetsMeta: SharedBetsMetadata?, 
    tagUserId: String?
) async throws -> CommentDTO

    變更後： 

    /// 發布新評論（頂層評論）
func publishComment(refId: String, content: String) async throws -> CommentDTO

/// 回覆現有評論
func publishReply(
    refId: String, 
    content: String, 
    parentId: Int64, 
    tagUserId: String, 
    sharedBetsMeta: SharedBetsMetadata?
) async throws -> CommentDTO

    原因： - 明確區分頂層評論和回覆的不同業務邏輯 - 簡化參數（publishComment 不需要 parentId 等可選參數） - 提高 API 可讀性和類型安全

    影響範圍： - PrematchCommentAPI+RepositoryProtocol.swift - PrematchCommentAPI+Repository.swift - PrematchCommentClient.swift - PrematchCommentRepository.swift

    向後兼容性：✅ 無影響（這是新實作）

    變更 1.3: toggleLike 返回值處理

    變更前： 

    func toggleLike(commentId: Int64) async throws -> CommentDTO?

    變更後： 

    func toggleLike(commentId: Int64) async throws  // 返回 void

    實作： 

    public func toggleLike(commentId: Int64) async throws {
    let endpoint = PrematchCommentEndpoint.toggleLike(commentId: commentId)
    // ⚠️ API 返回 raw null，使用 NullableResponse 處理
    let _: NullableResponse = try await apiClient.request(endpoint)
}

    原因： - 實際 API 返回 raw null，而非包裝的 DTO - 使用 NullableResponse 正確處理 null 值

    影響範圍： - PrematchCommentAPI+RepositoryProtocol.swift - PrematchCommentAPI+Repository.swift - PrematchCommentClient.swift - PrematchCommentRepository.swift

    向後兼容性：✅ 無影響（這是新實作）

    變更 1.4: 新增 NullableResponse DTO

    新增： 

    extension PrematchCommentAPI {
    /// 用於處理返回 null 的 API response（如 toggleLike）
    struct NullableResponse: Decodable {
        init(from decoder: Decoder) throws {
            // 允許解析 null 值，不做任何處理
            // 如果能成功到達這裡，代表 HTTP 請求成功
        }
    }
}

    原因： - 提供清晰的語意來處理返回 null 的 API - 避免 Decoding 錯誤

    2. TDD-011: ChatAPI

    變更 2.1: getBatchCount HTTP Method 修正

    變更前： 

    public var method: String {
    switch self {
    case .getBatchCount, .getChatroomInfo, .getHistoricalMessages:
        return FComHTTPMethod.get.rawValue
    // ...
    }
}

public var encoding: APIEncoding {
    switch self {
    case .getBatchCount, .getChatroomInfo, .getHistoricalMessages:
        return .url
    // ...
    }
}

    變更後： 

    public var method: String {
    switch self {
    case .getChatroomInfo, .getHistoricalMessages:
        return FComHTTPMethod.get.rawValue
    case .getBatchCount, .sendMessage, .joinChatroom, .leaveChatroom, .bulkLeaveChatroom:
        return FComHTTPMethod.post.rawValue  // ⚠️ getBatchCount 修正為 POST
    }
}

public var encoding: APIEncoding {
    switch self {
    case .getChatroomInfo, .getHistoricalMessages:
        return .url
    case .getBatchCount, .sendMessage, .joinChatroom, .leaveChatroom, .bulkLeaveChatroom:
        return .json  // ⚠️ getBatchCount 修正為 JSON encoding
    }
}

    原因： - 實際測試發現 API 只支援 POST 方法 - GET 方法返回 405 Method Not Allowed 錯誤

    影響範圍： - ChatAPI+Endpoint.swift - Integration Tests

    向後兼容性：✅ 無影響（這是新實作）

    變更 2.2: Null Response 處理

    新增： 

    extension ChatAPI {
    public actor ChatRepository: ChatRepositoryProtocol {
        public func joinChatroom(chatroomId: String) async throws {
            let endpoint = ChatEndpoint.joinChatroom(chatroomId: chatroomId)
            // ⚠️ API 返回 raw null，使用 NullableResponse 處理
            let _: NullableResponse = try await apiClient.request(endpoint)
        }

        public func leaveChatroom(chatroomId: String) async throws {
            let endpoint = ChatEndpoint.leaveChatroom(chatroomId: chatroomId)
            // ⚠️ API 返回 raw null，使用 NullableResponse 處理
            let _: NullableResponse = try await apiClient.request(endpoint)
        }

        public func bulkLeaveChatroom(chatroomId: String, excludeUserIds: [String]?) async throws {
            let endpoint = ChatEndpoint.bulkLeaveChatroom(chatroomId: chatroomId, excludeUserIds: excludeUserIds)
            // ⚠️ API 返回 raw null，使用 NullableResponse 處理
            let _: NullableResponse = try await apiClient.request(endpoint)
        }
    }

    /// 用於處理返回 null 的 API response
    struct NullableResponse: Decodable {
        init(from decoder: Decoder) throws {
            // 允許解析 null 值，不做任何處理
        }
    }
}

    原因： - joinChatroom、leaveChatroom、bulkLeaveChatroom API 實際返回 raw null - 使用 NullableResponse 正確處理這種情況

    影響範圍： - ChatAPI+Repository.swift

    3. TDD-031: LiveChatRepository

    變更 3.1: 兩層 Repository 架構

    變更前（原 TDD 規劃）： 

    Sources/LiveChat/Services/API/Chat/
  ├── ChatAPI+RepositoryProtocol.swift  # Repository Protocol
  ├── ChatAPI+Repository.swift          # Repository 實作（actor）
  └── ChatAPI+LiveChat.swift            # DTO → Domain Model 轉換

    變更後（實際實作）：

    API Layer (API Repository)： 

    Sources/MatchChat/Services/API/Chat/
  ├── ChatAPI+RepositoryProtocol.swift  # API Repository Protocol
  ├── ChatAPI+Repository.swift          # API Repository 實作（actor）
  └── ChatAPI+Models.swift              # DTO 定義
    - 職責：處理 HTTP 請求，返回 DTOs - 無業務邏輯

    Domain Layer (Domain Repository)： 

    Sources/MatchChat/Services/Repository/LiveChat/
  ├── LiveChatRepositoryProtocol.swift  # Domain Repository Protocol
  └── LiveChatRepository.swift          # Domain Repository 實作（actor）

Sources/MatchChat/Models/LiveChat/
  └── LiveChat+ChatAPI.swift            # DTO → Domain Model 轉換
    - 職責：使用 Client，將 DTOs 轉換為 Domain Models，包含業務邏輯 - 返回 Domain Models

    原因： 1. 職責分離： - API Repository 專注於網路請求 - Domain Repository 專注於業務邏輯和資料轉換

    1. 可測試性：
    2. API Repository 可以用 Mock APIClient 測試

    3. Domain Repository 可以用 Mock Client 測試

    4. 可維護性：

    5. API 變更只影響 API Repository

    6. Domain Model 變更只影響 Domain Repository

    7. 符合 Clean Architecture：

    8. API Repository 在內層（基礎設施層）
    9. Domain Repository 在外層（應用層）

    影響範圍： - 整體架構層次更清晰 - 測試策略更靈活

    向後兼容性：✅ 無影響（這是新實作）

    變更 3.2: WebSocket 訂閱包含黑名單過濾

    實作： 

    public func subscribeWebSocket(chatroomId: String) async throws -> AsyncStream<LiveChat.Message> {
    let webSocketStream = webSocketClient.subscribe(chatroomId: chatroomId)

    return AsyncStream { continuation in
        Task {
            for await webSocketMessage in webSocketStream {
                // 過濾黑名單用戶的訊息
                let userId = webSocketMessage.data.jsonBody.postUserId
                if await isBlacklisted(userId: userId) {
                    continue
                }

                let domainMessage = webSocketMessage.data.toDomainMessage()
                continuation.yield(domainMessage)
            }
            continuation.finish()
        }
    }
}

    原因： - 在 Repository 層統一處理黑名單邏輯 - 確保所有 WebSocket 訊息都經過黑名單過濾

    4. 架構文檔更新

    變更 4.1: 統一 MatchChat Package

    原 TDD 規劃： 

    Sources/
  ├── PrematchComment/  # 獨立 Package
  └── LiveChat/         # 獨立 Package

    實際實作： 

    MatchChat/Sources/MatchChat/  # 統一 Package
  ├── Models/
  │   ├── PrematchComment/
  │   └── LiveChat/
  └── Services/
      ├── API/
      ├── Client/
      └── Repository/

    原因： 1. 減少依賴複雜度：統一 Package 避免循環依賴問題 2. 共享代碼更容易：Models 和 Services 可以直接共享 3. 更符合實際專案結構：FCom 使用單一 Module 架構 4. 簡化構建和測試：單一 Package 更容易管理

    影響範圍： - 整體 Package 結構 - Import 語句統一為 import MatchChat

    向後兼容性：✅ 無影響（這是新實作）

    設計決策記錄

    1. 為何 API 直接返回 DTO/Array？

    決策：移除 Response Wrapper（bizCode/data 結構）

    理由： 1. 實際 API 行為：Backend 實際返回直接的 DTO/Array 2. 簡化代碼：減少不必要的包裝層和 Decoding 邏輯 3. 提高性能：減少一層 Object 創建

    替代方案考慮： - ❌ 保持 Wrapper：需要修改 Backend API（不現實） - ✅ 直接返回 DTO：符合實際情況，代碼更簡潔

    2. 為何拆分 publishComment？

    決策：拆分為 publishComment 和 publishReply

    理由： 1. 語意清晰：明確區分頂層評論和回覆 2. 類型安全：publishComment 不需要可選參數，減少錯誤 3. 符合業務邏輯：頂層評論和回覆的業務規則不同（如跨國回覆限制）

    替代方案考慮： - ❌ 單一方法 + 可選參數：容易誤用，語意不清 - ✅ 拆分方法：更清晰，更安全

    3. 為何採用兩層 Repository？

    決策：API Repository + Domain Repository

    理由： 1. 職責分離：API Repository 處理網路，Domain Repository 處理業務邏輯 2. 可測試性：兩層可以獨立測試 3. 可維護性：API 變更和業務邏輯變更互不影響 4. 符合 Clean Architecture：內層（API）和外層（Domain）分離

    替代方案考慮： - ❌ 單一 Repository：職責過重，難以測試 - ✅ 兩層 Repository：符合 SOLID 原則

    4. 為何統一 MatchChat Package？

    決策：使用統一的 MatchChat Package

    理由： 1. 避免循環依賴：獨立 Packages 容易產生循環依賴 2. 共享代碼更容易：Models 和 Services 可以直接共享 3. 符合專案結構：FCom 使用單一 Module 架構 4. 簡化管理：單一 Package 更容易構建和測試

    替代方案考慮： - ❌ 獨立 Packages：需要複雜的依賴管理 - ✅ 統一 Package：簡單直接，易於維護

    遷移指南

    對於未來的 UseCase 實作

    由於 API 和 Repository 的簽名已變更，UseCase 實作時需注意：

    1. 使用 publishComment vs publishReply

    // ✅ 發布頂層評論
let comment = try await repository.publishComment(
    refId: "sr:match:12345",
    content: "Great match!"
)

// ✅ 回覆評論
let reply = try await repository.publishReply(
    refId: "sr:match:12345",
    content: "I agree!",
    parentId: 123,
    tagUserId: "user123",
    sharedBetsMeta: nil
)

// ❌ 錯誤：不要使用舊的單一方法
// let comment = try await repository.publishComment(
//     refId: "sr:match:12345",
//     content: "Great match!",
//     parentId: nil,  // 容易誤用
//     sharedBetsMeta: nil,
//     tagUserId: nil
// )

    2. toggleLike 不返回值

    // ✅ 正確：不期待返回值
try await repository.toggleLike(commentId: "123")
// UI 狀態需要自己管理（例如樂觀更新）

// ❌ 錯誤：期待返回值
// let updatedComment = try await repository.toggleLike(commentId: "123")

    3. 使用 Domain Repository

    // ✅ 正確：使用 Domain Repository（返回 Domain Models）
let repository = PrematchCommentRepository(client: PrematchCommentClient())
let comments: [PrematchComment.Comment] = try await repository.getComments(...)
